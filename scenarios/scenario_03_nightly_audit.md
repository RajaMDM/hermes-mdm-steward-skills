# Scenario 03 — Scheduled nightly audit with email delivery

**Setup.** A data governance lead wants a DQ audit to run every weeknight at 23:00 UAE time. The audit should profile the supplier, product, and location master data, produce a summary, and email it to the on-duty steward so they have it waiting when they start the next morning.

This scenario shows how to wire the skill pack's `/mdm-dq-audit` skill into Hermes' built-in cron with the email gateway for delivery.

---

## What this scenario assumes

1. Hermes Agent is installed and running locally.
2. The email gateway is configured per the Hermes docs at `hermes-agent.nousresearch.com/docs/user-guide/messaging/`. For a Gmail user, this typically means an app password and SMTP credentials stored in `~/.hermes/.env`.
3. The `hermes-mdm-steward-skills` skill pack has been added to `~/.hermes/config.yaml` under `skills.external_dirs`.
4. `pandas` is available in the Hermes sandbox (it ships with most defaults).

## The conversation

```
User:
Set up a cron job to run the MDM DQ audit every weeknight at 23:00 UAE time
and email the summary to steward@nexora-retail.example at 07:30 the next
morning.

Agent:
Understood. Two separate schedules:
  - 23:00 weeknights: run /mdm-dq-audit against the bundled dataset, write
    the output to /tmp/dq_audit_summary.md.
  - 07:30 next weekday: email the generated summary to the steward.

I'll set up a single scheduled task that chains both — audit, then deliver
the briefing the next morning.

Proposed cron entries:
  - Audit:    "0 23 * * 1-5" (Mon–Fri at 23:00 local time)
  - Briefing: "30 7 * * 2-6" (Tue–Sat at 07:30 local time)

Before I create these, confirm:
  1. Local time zone: is this the machine local time (Asia/Dubai) or UTC?
  2. Email recipient: steward@nexora-retail.example — correct?
  3. Dataset: default to the bundled synthetic CSVs in the skill pack, or
     do you have a Core MDM export file path I should use instead?

User:
Asia/Dubai. Recipient correct. Use the bundled synthetic data for now — I
want to see the pipeline working before I point it at real exports.

Agent:
Creating the cron jobs now.

[agent uses Hermes' cron tool to register:]

Job 1 — mdm-nightly-audit
  schedule: 0 23 * * 1-5
  timezone: Asia/Dubai
  action:   invoke /mdm-dq-audit with default dataset paths
            output: /tmp/dq_audit_summary.md

Job 2 — mdm-morning-briefing
  schedule: 30 7 * * 2-6
  timezone: Asia/Dubai
  action:   invoke /mdm-steward-briefing
            consume: /tmp/dq_audit_summary.md
            delivery: email to steward@nexora-retail.example
            subject: "MDM Steward Briefing — {date}"

Both jobs are registered. You can inspect them with:
  hermes cron list

Run either manually to smoke-test without waiting for the schedule:
  hermes cron run mdm-nightly-audit
  hermes cron run mdm-morning-briefing

User:
Run both now as a smoke test.

Agent:
[agent triggers mdm-nightly-audit first, then mdm-morning-briefing.]

Audit complete. Summary written to /tmp/dq_audit_summary.md.

Key findings from this run:
  - 18 supplier records, 18 product records, 18 location records
  - 5 supplier duplicate candidates (shared TRN — confirmed match on all
    five pairs)
  - 6 product duplicate candidates (shared barcode across sub-brands —
    expected for shared-supplier products)
  - 6 emirate format violations (DXB / Dubayy variants on supplier and
    location records)
  - 3 cross-brand supplier overlaps (informational)

Briefing composed and dispatched to steward@nexora-retail.example.

Delivery receipt:
  - Message ID: <auto-generated by SMTP provider>
  - Sent: 2026-04-21 07:30:14 Asia/Dubai
  - Subject: "MDM Steward Briefing — 2026-04-21"

The briefing body was:

  Good morning.

  - 5 supplier duplicates confirmed via shared TRN. Two cross-brand
    (Verdant + Luxora). Merge clusters ready for approval in the queue.

    location records. Recommend a one-off cleanup pass on the emirate
```

## What this scenario demonstrates

1. **The skill pack composes into a scheduled pipeline** without custom orchestration code. Hermes' cron is the scheduler; the gateway is the delivery channel; the skills are the logic.
2. **Smoke-testing is first-class.** Any cron job can be invoked manually via `hermes cron run <job>`, so the pipeline is debuggable without waiting 24 hours.
3. **Separation of audit and briefing is deliberate.** The audit runs overnight (when the machine is idle); the briefing composes in the morning from yesterday's audit output. This means the briefing can be re-composed with human edits before sending, without re-running the audit.
4. **The output is a briefing, not a dashboard.** A steward gets an actionable email, not a wall of metrics. The full audit detail remains available at `/tmp/dq_audit_summary.md` for anyone who wants depth.

## Known limitations for local-only deployment

- **The machine must be running at 23:00 and 07:30.** If the machine is asleep or powered off at the scheduled times, the cron job does not fire.
- **If Hermes is not the active process** (e.g., the agent has not been launched since last boot), the cron daemon will not pick up the schedule.
- For a genuinely always-on deployment, move Hermes to a $5 VPS. The skill pack, the cron config, and the gateway setup all port cleanly — only the host changes.

## Extending this pipeline

- Swap the synthetic dataset for a nightly export from your real Core MDM platform. The script accepts arbitrary CSV paths — adjust the cron job's arguments.
- Add a second briefing recipient for a governance lead who wants a weekly roll-up (use a separate cron with a weekly frequency).
- Chain `/mdm-duplicate-resolver` and `/mdm-golden-record-composer` into the audit pipeline to pre-compute proposed merges for the steward's queue — so the morning briefing offers one-click approval on the clean-cut cases.
